# ITE Intelligence 2.0 — Architecture Blueprint

**Author:** Mikey Scholl + Claude
**Date:** March 14, 2026
**Status:** Design Complete — Ready for Build

---

## The Problem

The ITE Intelligence system has a fundamental connectivity issue: **260 PDF files in the guideline library cannot be reliably linked to their corresponding article records in the database.** The database contains 1,397 articles, each with a pre-computed `codon_filename` (format: `Author_Year#@#ART-XXXX@#@.pdf`), but zero of the 260 library PDFs actually use this naming convention.

This single disconnect has cascaded into:

- A 5-strategy matching cascade in the enricher (`lookup_article()`) that fails on ~25 files
- 67 crosswalk entries stuck in "partial" status
- Duplicated matching logic across 3 scripts (main enricher, batch enricher, crosswalk builder)
- Accumulated band-aids: keyword frequency filters, token-overlap thresholds, title stopword lists, apostrophe strippers — all compensating for unreliable filename-to-article matching

**Root cause:** The codon filename system was designed and populated in the database but never deployed to the actual PDF library. 184 of the 260 PDFs were collected in June 2025 (pre-project) with ad-hoc naming conventions (15+ distinct prefix patterns: `afp_`, `IDSA_`, `acg_`, `peds_`, `neuro_`, `rheum_`, `uspstf_`, `jacc_`, etc.).

---

## The Solution: Codon Migration

Rather than building permanent infrastructure to compensate for messy data, we fix the data at the source. Rename every PDF to its codon format, delete all derivative files, strip the band-aids from the pipeline, and re-process everything clean.

### What Survives (The Foundation)

The entire `ite_intelligence.db` database is preserved untouched:

| Asset | Count | What It Contains |
|-------|-------|-----------------|
| `articles` table | 1,397 rows | Tags, tiers, categories, blueprint mappings, citation counts, exam year linkages, priority scores, codon filenames, vector embeddings |
| `questions` table | 1,189 rows | ITE questions with tags, categories, answer data |
| `question_ref_pairs` table | 2,069 rows | Many-to-many QID ↔ article linkages with tier and exam year |
| `qid_art_xref` table | 1,818 rows | Reference crosswalk: QID ↔ ART-ID with tier, exam year, author, year |
| `article_vec` / `question_vec` | 1,397 / 1,189 | Vector embeddings for semantic search |

This is the intelligence layer. It took months to build. It stays.

### What Gets Deleted (Derivative Files)

All derivative files are regenerated by the pipeline — they are outputs, not source data:

- Enriched JSONs (`03_enriched JSON/`) — 228 files
- DOCX summaries (`04_DOCX_summaries/`)
- ITE reference JSONs (`05_ite_ref_json/`)
- Crosswalk index (`crosswalk_index.json`)

### What Gets Trimmed (Code Band-Aids)

After migration, Strategy 0 (codon parse) handles 100% of files. The following code becomes dead weight:

| Component | What Gets Removed | Why It Existed |
|-----------|-------------------|----------------|
| `lookup_article()` Strategies 2-5 | Author+year guessing, keyword frequency analysis, org pattern matching, vector similarity fallback | Compensated for unreliable filenames |
| `_word_freq_in_db()` | Keyword frequency filter (threshold ≤10) | Tried to avoid common-word false positives |
| `_TITLE_STOPWORDS` | 50+ stopwords for title matching | Made fuzzy title matching slightly less bad |
| `_strip_apostrophes()` | Apostrophe normalization | Edge case in title matching |
| `_title_keywords()` | Title keyword extraction | Fed into Strategy 3 |
| Crosswalk 3-pass matching | Token overlap, slug normalization, threshold tuning (300 lines) | Tried to match PDFs to derivatives without DB awareness |
| Duplicated `lookup_article()` | Identical function in both `ite_intelligence_enricher.py` and `ite_intelligence_enricher_batch.py` | Copy-paste instead of shared module |

**Post-migration `lookup_article()`** becomes approximately:

```python
def lookup_article(conn, source_block, json_path):
    """Parse ART-ID from codon filename. That's it."""
    file_name = source_block.get("file_name", "") or json_path.stem
    match = re.search(r'#@#(ART-\d+)@#@', file_name)
    if match:
        art_id = match.group(1)
        cur = conn.cursor()
        cur.execute("SELECT clean_ref, tier FROM articles WHERE article_id=?", (art_id,))
        row = cur.fetchone()
        if row:
            return _build_payload(conn, row[0], row[1], "codon_filename")
    return None  # fallback: log for manual review
```

## Migration Plan — 8 Steps

### Step 1: Build Match Folders (Dry Run)

**Script:** `build_match_staging.py`
**Input:** `crosswalk_index.json` + `ite_intelligence.db`
**Output:** Dry-run report showing proposed folder pairings

The script reads the crosswalk and DB, then proposes a folder structure:

```
match_staging/
├── ART-0612_COPD/
│   ├── afp_COPD_dx_mgmt.pdf
│   ├── 218_afp_COPD_dx_mgmt_extracted.json
│   └── ART-0612_COPD_summary.docx
├── ART-0487_Gentry_2017/
│   ├── afp_fatigue_eval.pdf        ← WAIT, wrong match?
│   └── ...
└── _unmatched/
    ├── periop_bloodsugar_mngmt.pdf
    └── ...
```

**Three match tiers:**

| Tier | Count (est.) | Signal | Confidence |
|------|-------------|--------|------------|
| Tier 1: Full match | ~192 | Crosswalk "full coverage" — enriched JSON has `ite_intelligence` block with ART-ID | High — JSON *is* the key |
| Tier 2: Partial match | ~43 | Enriched JSON exists but missing `ite_intelligence` block | Medium — has title/year/author, needs resolution |
| Tier 3: No derivatives | ~25 | No enriched JSON at all | Low — filename + PDF metadata only |

Dry-run mode: generates the report, moves nothing.

### Step 2: Resolve Tier 2 Mismatches

**Method:** Claude API batch call (using existing Anthropic Message Batches API setup)
**Input:** Each unmatched file's `source.title`, `source.publication_year`, `source.authors` from enriched JSON
**Query against:** `articles` table (title, author1, author2, year, clean_ref)
**Fallback:** Vector similarity using `article_vec` embeddings

For Tier 3 (no enriched JSON): extract metadata from PDF filename patterns + first page text, then match.

**Output:** Proposed ART-ID for each unmatched file, with confidence score.

This is a good candidate for running on your local API key — it's a batch of ~43-68 calls, low cost, and you can review the results at your own pace.

### Step 3: Human Review Checkpoint

**Nothing irreversible has happened yet.**

Mikey opens the match folders (or reviews the dry-run report), confirms pairings, flags any bad matches. Key things to eyeball:

- Tier 1 matches that look wrong (false positives from crosswalk)
- Tier 2 proposed matches that don't make clinical sense
- Any PDFs that should be discarded (duplicates, outdated, irrelevant)

Bad matches get moved to `_needs_review/`. Confirmed matches proceed.

### Step 4: Rename PDFs to Codon Format

**Script:** `rename_to_codon.py`
**Input:** Confirmed match folders
**Source of truth:** `articles.codon_filename` field in the DB

The script walks each match folder, reads the ART-ID from the folder name, queries the DB for the pre-computed `codon_filename`, and renames the PDF.

Example: `afp_COPD_dx_mgmt.pdf` → `AFP_COPD_2017#@#ART-0612@#@.pdf`

**Safety features:**
- `--dry-run` by default (shows what would change)
- `--execute` flag required to actually rename
- Generates `rename_log.json` for full rollback capability
- Validates that target filename doesn't already exist

### Step 5: Flatten and Clean

- Move all renamed PDFs back to `01_pdf_guideline_library/`
- Delete all files in `03_enriched JSON/`, `04_DOCX_summaries/`, `05_ite_ref_json/`
- Delete `crosswalk_index.json`
- Archive `rename_log.json` for provenance

### Step 6: Trim the Pipeline

Simplify the codebase:

- **`ite_intelligence_enricher.py`**: Replace 5-strategy `lookup_article()` with codon-only Strategy 0 + simple fallback logging
- **`ite_intelligence_enricher_batch.py`**: Same simplification (or import from shared `article_resolver.py` if preferred)
- **`build_crosswalk_index.py`**: Rewrite to parse ART-IDs from filenames — ~50 lines instead of 300
- **Optional:** Create `article_resolver.py` as shared module if we want one import point

### Step 7: Full Re-Process

Run the complete pipeline on the clean, codon-named library:

```
PDF (codon-named) → extract (main.py) → synthesize (synthesize.js) → enrich (enricher.py) → DOCX (build_summary.js) → save JSON
```

Every file hits Strategy 0, matches instantly, pulls tagging data from DB, generates fresh derivatives.

**Benefit of re-processing:** Earlier extractions used older prompts and pipeline versions. A clean extraction with the current pipeline gives consistent quality across the entire library.

### Step 8: Verify Integrity

- Confirm all 260 PDFs have codon names
- Confirm enricher matched 260/260 (Strategy 0, zero fallbacks)
- Confirm crosswalk shows 260/260 full coverage
- Confirm DB tagging data is intact (spot-check high-priority articles)
- Confirm `qid_art_xref` still resolves correctly
- Generate new `crosswalk_index.json` with 100% coverage

## Intelligence 2.0 — Four New Layers

With the migration complete and the pipeline clean, the database becomes the foundation for a clinical knowledge intelligence system that extends far beyond ITE exam prep.

### Layer 1: ICD-10 Diagnostic Linkage

**New table:** `article_icd10`

```sql
CREATE TABLE article_icd10 (
    article_id  TEXT NOT NULL,
    icd10_code  TEXT NOT NULL,
    icd10_desc  TEXT,
    relevance   TEXT,  -- 'primary', 'secondary', 'related'
    PRIMARY KEY (article_id, icd10_code)
);
```

**What it does:** Tags each article with its relevant ICD-10 diagnosis codes. Transforms the library from exam-category organization (how the ITE thinks) to diagnosis-based organization (how a clinician thinks).

**How to build:** Claude API batch pass — send each article's title + categories + clean_ref, ask for top 3-5 ICD-10 codes. Validate against the ICD-10 MCP (already connected). Estimated: single batch call, ~1,397 items.

**What it unlocks:**
- "Show me everything related to E11 (Type 2 DM)" pulls articles across categories — endocrine management, cardiovascular risk, retinopathy screening, nephropathy monitoring, podiatric care
- "Which diagnoses in my library have the most exam coverage?" — cross-reference with citation counts
- Foundation for Layer 3 (Clinical Pathways)

### Layer 2: Article Currency via PubMed

**New table:** `article_currency`

```sql
CREATE TABLE article_currency (
    article_id          TEXT PRIMARY KEY,
    pubmed_id           TEXT,
    last_checked        TEXT,  -- ISO date
    newer_version_pmid  TEXT,
    newer_version_date  TEXT,
    newer_version_title TEXT,
    currency_status     TEXT   -- 'current', 'superseded', 'updated', 'check_needed'
);
```

**What it does:** Tracks whether each article in the library is still the most current version of that guideline. Queries PubMed for newer publications by the same authors/organizations on the same topic.

**How to build:** PubMed MCP (already connected) — for each article, search by author + key title terms + date range (article year → present). Flag any newer publications with matching topic signatures.

**What it unlocks:**
- "You're studying Gentry 2017, but there's a 2024 update" — proactive currency alerts
- `superseded_by` chain: old article → newer version → newest version
- Identify which clinical pathways (Layer 3) have stale components
- Scheduled refresh: run monthly to keep the library current

### Layer 3: Clinical Pathway Engine

**New table:** `clinical_pathways`

```sql
CREATE TABLE clinical_pathways (
    article_id    TEXT NOT NULL,
    icd10_code    TEXT NOT NULL,
    pathway_role  TEXT NOT NULL,  -- 'screening', 'diagnosis', 'first_line_tx', 'monitoring', 'referral', 'prevention', 'patient_education'
    source_org    TEXT,           -- 'USPSTF', 'ADA', 'ACC/AHA', 'AFP', etc.
    confidence    REAL,
    PRIMARY KEY (article_id, icd10_code, pathway_role)
);
```

**What it does:** This is the clinical blending engine. For any given diagnosis, it assembles the complete evidence chain from multiple sources — USPSTF screening recs, society diagnostic criteria, AFP treatment guidelines, specialist referral thresholds — into a single cohesive reference organized by clinical workflow stage.

**Example — Type 2 Diabetes (E11):**

| ART-ID | Source Org | Pathway Role | What It Covers |
|--------|-----------|--------------|----------------|
| ART-0234 | USPSTF | screening | When to screen, risk factors, A1c thresholds |
| ART-0891 | ADA | diagnosis | Diagnostic criteria, A1c vs. fasting glucose |
| ART-0445 | AFP | first_line_tx | Metformin initiation, lifestyle modifications |
| ART-1102 | ACC/AHA | monitoring | ASCVD risk calculation, statin initiation criteria |
| ART-0672 | AAO | referral | Retinopathy screening timeline, ophthalmology referral |
| ART-0389 | ADA | monitoring | Annual microalbumin, nephropathy monitoring |

No single guideline covers the whole patient. AFP tells you what to prescribe. USPSTF tells you when to screen. ACC/AHA tells you when to add a statin. This table assembles the fragments into a complete clinical picture, indexed by diagnosis.

**How to build:** Builds on Layer 1 (ICD-10 tags). Claude API batch pass — send each article + its ICD-10 codes, ask "what role does this article play in the clinical management pathway for [diagnosis]?" Requires clinical review (Mikey eyeballs the assignments).

**What it unlocks:**
- Personal UpToDate: "Patient has E11 → here's your evidence chain, organized by care stage"
- Gap analysis: "You have screening and diagnosis covered for E11, but no referral criteria article"
- When Layer 2 flags a superseded article, Layer 3 identifies exactly which pathway role has a gap
- Study organization by clinical workflow, not just exam category

**Build priority:** Most ambitious layer. Build after Layers 1 and 2 are mature and ICD-10 tags are validated.

### Layer 4: Predictive Trend Intelligence + Alert System

**New table:** `topic_trends`

```sql
CREATE TABLE topic_trends (
    category        TEXT NOT NULL,
    exam_year       INTEGER NOT NULL,
    question_count  INTEGER,
    article_count   INTEGER,
    avg_tier_weight REAL,
    PRIMARY KEY (category, exam_year)
);
```

**New table:** `pubmed_alerts`

```sql
CREATE TABLE pubmed_alerts (
    alert_id        INTEGER PRIMARY KEY AUTOINCREMENT,
    pubmed_id       TEXT,
    title           TEXT,
    authors         TEXT,
    pub_date        TEXT,
    journal         TEXT,
    alert_trigger   TEXT,   -- 'trending_topic', 'key_author', 'key_journal', 'high_citation'
    relevance_score REAL,
    matched_category TEXT,
    status          TEXT,   -- 'new', 'reviewed', 'added_to_library', 'dismissed'
    created_date    TEXT
);
```

**What it does (Trends):** Aggregates exam question frequency by category across years (2020-2024). Identifies which topics are increasing, stable, or declining in ITE emphasis. Cross-references with PubMed publication volume for the same topics.

**What it does (Alerts):** Automated new-article detection. Queries PubMed on a schedule for new publications matching criteria derived from historical ITE data:

- **Key journals:** Journals that appear most frequently in ITE references (Am Fam Physician, NEJM, JAMA, etc.)
- **Key authors:** Authors with high citation counts in the DB
- **Trending topics:** Categories showing upward year-over-year exam frequency
- **Guideline updates:** New publications from USPSTF, ADA, ACC/AHA, ACOG, AAP, etc.

When a new publication matches multiple criteria — published in a key journal, on a trending topic, by a frequently-cited author — it gets a high relevance score and triggers an alert:

> **ALERT: Possible ITE Article**
> "Updated Guidelines for Management of Chronic Heart Failure" — ACC/AHA, JACC 2026
> **Why:** Heart failure trending ↑40% in ITE questions (2022→2024). ACC/AHA is source org for 12 existing articles. Supersedes ART-0445 (2022 guidelines).
> **Action:** [Add to library] [Dismiss] [Review later]

**How to build:**
- `topic_trends`: Single SQL aggregation from existing `qid_art_xref` + `articles` data — can build today
- `pubmed_alerts`: Scheduled PubMed API queries (weekly/monthly) using criteria extracted from DB patterns
- Alert delivery: Could be a scheduled Cowork task, email via Gmail MCP, or Google Calendar reminder via GCal MCP

**What it unlocks:**
- Predictive study targeting: "Cardiovascular pharmacology is up 35% over 3 years — prioritize these articles"
- Proactive library maintenance: new guidelines get flagged before you have to go looking for them
- Longitudinal insight: track how ABFM's exam emphasis shifts over time
- When both signals align (exam frequency up AND new guidelines being published), that's a high-confidence prediction of next year's ITE content

## How the Layers Connect

The four layers are not independent — they form one integrated system:

```
Layer 1 (ICD-10)        → Connects articles to diagnoses
     ↓
Layer 3 (Pathways)      → Assembles articles into clinical logic per diagnosis
     ↓
Layer 2 (PubMed)        → Monitors currency of each article in each pathway
     ↓
Layer 4 (Trends/Alerts) → Predicts what's coming and flags new evidence
     ↓
     → Feeds back into Layer 1 (new article gets ICD-10 tags)
     → Feeds back into Layer 3 (new article fills a pathway gap)
```

**The cycle:**

1. Alert system detects a new guideline publication (Layer 4)
2. You add it to the library with a codon filename
3. Pipeline extracts and enriches it automatically (Strategy 0 match)
4. ICD-10 tagging assigns diagnostic codes (Layer 1)
5. Clinical pathway engine identifies which care pathway it serves (Layer 3)
6. Currency tracker marks the old version as superseded (Layer 2)
7. Trend data updates with the new article's category (Layer 4)

The system maintains itself.

## Key Design Principles

1. **Fix the data, not the code.** The codon migration eliminates the need for complex matching infrastructure. When the naming is right, the code is simple.

2. **Each layer is just a table and a query.** No new pipeline steps, no new scripts to maintain. Intelligence 2.0 features are data layers on a clean foundation.

3. **Preprocessing over runtime computation.** Everything deterministic happens up front (ICD-10 tagging, trend aggregation, currency checks) so the expensive/intelligent steps only handle what actually requires intelligence.

4. **Human checkpoints before irreversible actions.** Match folders are reviewable. Dry-run is default. Rename logs enable rollback.

5. **Modular and additive.** Each layer can be built, tested, and validated independently. Adding a new layer never requires modifying an existing one.
